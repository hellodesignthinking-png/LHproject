## 🔴 **솔직한 v4.1 문제 인정**

### **확실히 완료된 것 (Good)**
- ✅ 기술 아키텍처 안정적
- ✅ 6종 보고서 엔드포인트 작동
- ✅ 기본 구조 (섹션, 해석, GO/NO-GO)
- ✅ M2-M6 데이터 파이프라인

### **실제로는 미완성인 것 (Bad - 이번에 고칠 것)**

#### **🔴 Problem 1: 페이지 밀도 불균형**
**현재**:
- All-in-One / LH Technical: 50-60p
- Financial / Quick / Presentation: 30-40p

**문제**:
- "모두 60페이지급" 메시지와 실물 불일치
- 보고서별 완성도 편차 존재

**v4.2 목표**:
- **모든 보고서 최소 50페이지 이상**
- 장(Chapter)당 최소 3페이지

---

#### **🔴 Problem 2: 정책·제도 파트 얕음**
**현재**:
- 정책 설명 존재
- 하지만 "왜 이 정책이 작동하는지" 구조 설명 얕음

**예시**:
- LH 신축매입임대: 개요 수준
- 심사 기준: 항목 나열 수준

**문제**:
- 외부 심사관/공공기관 관점에서 **"근거 서술 부족"**

**v4.2 해결책**:
정책·제도 장에 **5가지 필수 요소** 추가:
1. **제도의 도입 배경** (왜 생겼나?)
2. **제도가 해결하려는 정책 문제** (목적은?)
3. **실제 LH 심사·집행 구조** (어떻게 작동하나?)
4. **대상 사업에 미치는 영향** (우리 프로젝트엔?)
5. **제도상 한계와 비공식 관행** (실제 운영은?)

---

#### **🔴 Problem 3: 리스크 파트 형식적**
**현재**:
- 리스크: 정책 리스크, 사업성 리스크, 구조적 리스크
- 대부분 **1-2페이지**
- "있어 보이지만 실제 의사결정엔 부족"

**문제**:
- 리스크 나열만 있고 **깊이 없음**
- 대응 전략이 추상적

**v4.2 해결책**:
리스크별 **3단 구조** 강제:

```
① 발생 원인 (왜 발생하나?)
  - 근본 원인 분석
  - 트리거 조건
  
② 실제 발생 가능한 상황 (현실 사례)
  - 유사 프로젝트 사례
  - 구체적 시나리오
  
③ 대응 전략 및 완화 가능성
  - 사전 예방 방안
  - 발생 시 대응책
  - 완화 가능성 평가
```

**리스크 하나당 최소 1페이지 이상**

---

#### **🔴 Problem 4: 시나리오 분석 설득력 부족**
**현재**:
- 시나리오 장 존재
- 하지만 **수치 변화에 따른 판단 변화 설명 부족**

**문제**:
- "그래서 뭐가 달라지는지" 불명확
- 투자자·LH 모두에게 설득력 부족

**v4.2 해결책**:
**3개 시나리오 필수**:
1. **기준 시나리오** (Base Case)
2. **보수적 시나리오** (Worst Case)
3. **적극적 시나리오** (Best Case)

각 시나리오마다 **3가지 명시**:
- **수치 변화**: NPV, IRR, ROI 등 핵심 지표 변화
- **판단 변화**: GO → REVIEW 또는 REVIEW → NO-GO 등
- **의사결정 결과 변화**: 투자 규모, 타이밍, 조건 변경

---

#### **🔴 Problem 5: 프레젠테이션 보고서 슬라이드 최적화 미흡**
**현재**:
- 페이지 분리 존재
- 메시지 강조 있음

**문제**:
- 여전히 "문서 → 슬라이드 변환" 느낌
- **발표자가 읽기엔 문장 길고 설명형**

**v4.2 해결책**:
**프레젠테이션 전용 규칙**:

```
규칙 1: 한 페이지 = 한 핵심 메시지
규칙 2: 설명은 '발표 스크립트 톤'
규칙 3: 문장 길이 2줄 초과 금지
규칙 4: "그래서 결론은?"이 항상 페이지 하단 위치
규칙 5: 슬라이드 번호 + 진행 표시 (예: 3/14)
```

**슬라이드 구조 강제**:
```
┌──────────────────────────────┐
│ [핵심 메시지 1줄]             │ <- 제목
│                              │
│ • 근거 1 (1줄)               │ <- 본문 (최대 3개 불릿)
│ • 근거 2 (1줄)               │
│ • 근거 3 (1줄)               │
│                              │
│ → 결론: [판단/행동] (1줄)    │ <- 하단 결론
└──────────────────────────────┘
```

## 📋 **v4.2 구현 체크리스트**

### **Phase 1: 분량 확보 (All Reports 50p+)**
- [ ] All-in-One 보고서: 현재 60p → 유지
- [ ] Landowner 보고서: 현재 40p → **50p 확대**
- [ ] LH Technical 보고서: 현재 40p → **50p 확대**
- [ ] Financial Feasibility: 현재 50p → 유지
- [ ] Quick Check: 현재 50p → 유지
- [ ] Presentation: 현재 55p → 유지 (단, 슬라이드 최적화)

### **Phase 2: 정책·제도 강화 (All Reports)**
각 보고서의 정책 장에 추가:
- [ ] 제도 도입 배경 (1-2 단락)
- [ ] 정책 문제 정의 (1-2 단락)
- [ ] LH 심사 구조 상세 (2-3 단락)
- [ ] 프로젝트 영향 분석 (2-3 단락)
- [ ] 한계·관행 서술 (1-2 단락)

### **Phase 3: 리스크 고도화 (All Reports)**
리스크별 3단 구조 적용:
- [ ] 정책 리스크: 원인 → 사례 → 대응 (3p)
- [ ] 사업성 리스크: 원인 → 사례 → 대응 (3p)
- [ ] 구조적 리스크: 원인 → 사례 → 대응 (3p)
- [ ] 최소 9페이지 확보 (3개 리스크 × 3p)

### **Phase 4: 시나리오 고정 (Financial, All-in-One)**
- [ ] 기준 시나리오: 수치 + 판단 + 결과
- [ ] 보수 시나리오: 수치 + 판단 + 결과
- [ ] 적극 시나리오: 수치 + 판단 + 결과
- [ ] 시나리오 비교 표 추가
- [ ] 최소 6페이지 확보

### **Phase 5: 프레젠테이션 최적화**
- [ ] 모든 슬라이드: 1핵심 메시지
- [ ] 문장 길이: 2줄 제한 강제
- [ ] 결론 위치: 페이지 하단 고정
- [ ] 슬라이드 번호: X/총개수 표시
- [ ] 발표 스크립트 톤 적용

### **Phase 6: 품질 검증**
- [ ] 모든 보고서 50페이지 이상 확인
- [ ] 정책 근거 5가지 요소 존재 확인
- [ ] 리스크 3단 구조 존재 확인
- [ ] 시나리오 3가지 + 판단 변화 확인
- [ ] 프레젠테이션 슬라이드 규칙 준수 확인
