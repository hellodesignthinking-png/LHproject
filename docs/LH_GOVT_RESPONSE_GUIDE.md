# 🏛️ ZeroSite 4.0 LH/지자체 대응 가이드

**Version**: 1.0  
**Date**: 2025-12-27  
**Target**: 영업/고객 대응팀

---

## 🎯 대응 가이드 목적

**핵심 원칙:**
1. ZeroSite 4.0은 **판단을 내리는 시스템**이 아닌 **판단을 지원하는 시스템**
2. 최종 판단 권한은 **항상 LH/지자체에 있음**
3. 시스템은 **객관적 데이터와 분석**만 제공

---

## 📋 1️⃣ 자주 묻는 질문 (FAQ)

### Q1: "ZeroSite는 어떻게 판단하나요?"

**✅ 올바른 답변:**
```
ZeroSite 4.0은 LH 공사의 공식 심사 기준을 기반으로
객관적인 데이터를 분석하여 점수를 산출합니다.

5개 영역(정책, 입지, 건설, 가격, 사업성)을 종합하여
총 100점 만점으로 평가하며, 70점 이상 시 '조건부 가능' 이상으로 판단됩니다.

다만, 최종 승인 여부는 LH 공사의 심사위원회에서 결정합니다.
```

**❌ 피해야 할 답변:**
- "저희 AI가 자동으로 판단합니다"
- "90% 확률로 승인됩니다"
- "LH보다 정확합니다"

---

### Q2: "점수 계산 방식을 공개할 수 있나요?"

**✅ 올바른 답변:**
```
점수 계산은 LH 공사의 공식 심사 기준을 기반으로 하며,
5개 섹션의 가중 평균으로 산출됩니다:

- 정책 적합성 (30점)
- 입지 분석 (25점)
- 건설 가능성 (20점)
- 가격 적정성 (15점)
- 사업성 (10점)

세부 알고리즘은 영업 기밀이나,
보고서에는 각 섹션별 점수와 감점/가점 사유가
모두 명시됩니다.
```

**❌ 피해야 할 답변:**
- "알고리즘은 비공개입니다" (거부 톤)
- "소스 코드를 보시면 알 수 있습니다" (불가능한 제안)

---

### Q3: "왜 CONDITIONAL 판정이 나왔나요?"

**✅ 올바른 답변:**
```
CONDITIONAL 판정은 다음을 의미합니다:

1. 현재 조건에서는 일부 개선이 필요하나,
2. 개선 가능한 항목이 명확히 식별되었으며,
3. 개선 시 LH 매입 가능성이 있음을 나타냅니다.

보고서의 '개선 포인트' 섹션에서
- 개선 필요 항목
- 예상 점수 증가분
- 구체적 개선 방안
을 확인하실 수 있습니다.

이는 부정적 평가가 아닌, '개선 로드맵'을 제공하는 것입니다.
```

**❌ 피해야 할 답변:**
- "잘 모르겠습니다"
- "점수가 낮아서요"
- "재심사 받으세요"

---

### Q4: "다른 시스템과 결과가 다른데요?"

**✅ 올바른 답변:**
```
ZeroSite 4.0은 다음 점에서 차별화됩니다:

1. **Single Source of Truth**
   - 하나의 통합된 판단 기준(M6)만 사용
   - 보고서 간 판단 불일치 구조적 차단

2. **Kill-Switch 시스템**
   - 사람의 주관적 개입 자동 차단
   - 일관성 자동 검증

3. **LH 공식 기준 준수**
   - LH 공사 심사 기준 기반
   - 실제 승인 사례 데이터 반영

다른 시스템은 이러한 검증 체계가 없을 수 있으며,
결과 차이는 자연스러운 현상입니다.

최종 판단은 LH 공사가 하므로,
여러 시스템의 결과를 참고하시되
공식 심사를 진행하시기 바랍니다.
```

**❌ 피해야 할 답변:**
- "저희가 더 정확합니다"
- "다른 시스템이 틀렸습니다"
- "우리 것만 믿으세요"

## 🏛️ 2️⃣ LH 공사 대응

### 상황 1: LH 담당자가 점수에 이의 제기

**대응 시나리오:**

```
담당자: "주차 효율 점수가 너무 낮게 나온 것 같은데요?"

✅ 올바른 대응:
"주차 효율 점수는 다음 기준으로 산출되었습니다:
- 법정 주차 대수: [X]대
- 실제 계획 주차: [Y]대
- 주차 효율: [Z]%

LH 기준상 주차 효율 90% 이상이 권장되며,
현재 [Z]%는 기준 미달로 -4점 감점 처리되었습니다.

개선 방안:
1. 기계식 주차 도입 (+2점)
2. 지하 주차장 확장 (+4점)

재평가를 원하시면 개선 계획 반영 후
재분석을 요청해주시기 바랍니다."

❌ 피해야 할 대응:
"시스템이 그렇게 나왔습니다"
"알고리즘을 수정해드릴 수 없습니다"
```

---

### 상황 2: LH가 다른 결과를 제시

**대응 시나리오:**

```
LH: "저희 내부 검토 결과는 65점인데, 귀사는 75점이네요?"

✅ 올바른 대응:
"차이가 발생할 수 있는 원인은 다음과 같습니다:

1. 평가 시점 차이
   - ZeroSite: [날짜]
   - LH 검토: [날짜]
   → 시장 변동 반영 차이

2. 데이터 소스 차이
   - ZeroSite: [출처]
   - LH: 내부 데이터
   → 기준 시점 상이

3. 평가 항목 가중치 차이
   - ZeroSite: LH 공식 기준 기반
   - LH: 심사위원회 재량 반영

최종 판단은 LH 심사위원회가 하시므로,
ZeroSite 결과는 참고용으로만 활용해주시기 바랍니다.

필요 시 데이터 상세 비교 및 재분석을
지원해드리겠습니다."

❌ 피해야 할 대응:
"LH가 틀렸을 수도 있습니다"
"저희 시스템이 더 최신입니다"
```

---

### 상황 3: 보고서 공식 제출 요청

**대응 시나리오:**

```
LH: "이 보고서를 공식 심사 자료로 제출해도 될까요?"

✅ 올바른 대응:
"네, 가능합니다. 다만 다음 사항을 확인해주세요:

1. 제출용 보고서 선택
   - LH 기술검토 보고서 (권장)
   - 사업타당성 보고서 (추가 제출 가능)

2. 제출 전 검증
   - 개인 정보 익명화 확인
   - 최신 데이터 반영 확인
   - 유효 기간 확인 (생성일 기준 3개월)

3. 제출 패키지 구성
   - 공식 보고서 PDF
   - 데이터 요약 Excel
   - 메타데이터
   - 사용 안내서

제출 패키지 생성을 도와드리겠습니다."

❌ 피해야 할 대응:
"마음대로 사용하세요"
"책임은 안 집니다"
```

### 상황 2: 정책 변경으로 재분석 요청

**대응 시나리오:**

```

"네, 정책 변경 반영이 가능합니다:

1. 변경 내용 확인
   - 변경 항목: [상세]

2. 시스템 반영 절차

   - E2E 테스트 재실행
   - 예상 소요: 3~5일

3. 영향 받는 사업지
   - 기존 평가 대상: [N]건

   - 일괄 재평가 지원

정책 변경은 시스템 안정성 확인 후
적용됩니다."

❌ 피해야 할 대응:
"시스템 수정은 어렵습니다"
"다음 버전에서 반영됩니다" (구체적 일정 없이)
```

## 📞 5️⃣ 에스컬레이션 절차

### Level 1: 일반 문의 (고객센터)

**처리:**
- FAQ 기반 답변
- 보고서 해석 지원
- 데이터 확인

---

### Level 2: 기술 문의 (기술 지원팀)

**처리:**
- 점수 계산 상세 설명
- 데이터 소스 검증
- 재분석 요청 처리

**에스컬레이션 조건:**
- LH/지자체 공식 요청
- 판정 이의 제기
- 시스템 오류 의심

---

### Level 3: 긴급 대응 (개발팀)

**처리:**
- Kill-Switch 재검증
- E2E 테스트 재실행
- M6 판단 로직 검증
- 긴급 패치 적용

**에스컬레이션 조건:**
- Kill-Switch FAILED
- M6 일관성 깨짐
- 중대한 시스템 오류

---

## 🎯 대응 원칙 요약

### DO ✅

1. **객관적 데이터 제시**
   - 점수 산출 근거 명확히
   - 출처 명시
   - 개선 방안 구체적 제시

2. **LH/지자체 최종 판단 존중**
   - "참고용으로 활용"
   - "최종 판단은 OO에서"
   - "공식 심사 진행 권장"

3. **개선 가능성 강조**
   - CONDITIONAL은 기회
   - NOGO도 대안 제시
   - 구체적 로드맵 제공

---

### DON'T ❌

1. **절대적 표현 금지**
   - "100% 승인됩니다"
   - "절대 안 됩니다"
   - "우리가 LH보다 정확합니다"

2. **다른 시스템 비하 금지**
   - "다른 시스템은 틀렸습니다"
   - "우리만 믿으세요"

3. **책임 회피 금지**
   - "시스템이 그렇게 나왔습니다"
   - "모르겠습니다"
   - "불가능합니다" (대안 없이)
